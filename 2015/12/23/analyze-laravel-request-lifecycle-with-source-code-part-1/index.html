  • laravel:5.1
  • 项目:干净的helloworld
    整个laravel框架的入口文件位于 /project_name/public/index.php

篇幅有限，我删除了代码注释，大家可以看看3大步骤的注释，挺有趣的

我们一行一行来

首先入口文件 引入了bootstrap文件夹下的autoload.php，注释说得很清楚，这是composer生成的类加载器，避免了手动加载带来的麻烦，其实不然，我们跟到autoload.php一探究竟

autoload.php首先定义了整个框架启动的时间，应该用于日志记录或者debug，这个不再多说，下面引入的vendor下的autoload.php才是composer生成的类加载器，如果还想打破砂锅问到底，我良辰奉陪到底

autoload_real.php就是ComposerAutoloaderInit231cc0decba412cbd019c190c91f0196这个类，之所以名字这么长应该是通过一定的hash算法来确保唯一性，在每次执行composer install安装依赖或者update升级依赖后都会重新生成一个名为ComposerAutoloaderInit*的类

好我们继续回到auto_load.php文件下
`$compiledPath = DIR.’/cache/compiled.php’;

接着它判断了“编译”后的类文件是否存在，存在则引用进来，这个compiled.php是干嘛的呢？

先看看注释

To dramatically increase your application’s performance, you may use a compiled class file which contains all of the classes commonly used by a request. The Artisan “optimize” is used to create this file.
在项目正式上线之前项目的文件都是不确定的，在项目文件不再变动时确认上线时我们可以在项目根目录执行
php artisan optimize
artisan就会生成优化后的类加载器，大大提高加载速度

不扯远了，回到入口文件

我们现在来看看bootstrap下的app.php是怎样返回$app的

因为这是part 1 我就不再跟进Illuminate\Foundation\Application了，$app怎么来的我会在以后的系列中分析，通过Application的构造函数，我们拿到了$app这个实例
通过函数名不难得出，$app又通过单例模式绑定了HTTP Kernel & Console Kernel & ExceptionHandler三大核心组建，说得过去，laravel是个web框架，肯定要处理http请求，HTTP Kernel则包含了处理http请求的所有逻辑，同样Console Kernel则处理来自CLI的请求，当然不可能每次请求都是一帆风顺的，所以最后$app喊上了异常处理，至此$app已经武装到了牙齿，圆润地滚回了入口文件

我们也暂时不谈具体实现，看看$app回到入口文件又干了哪些工作

$app通过make方法拿到kernel的实例，Illuminate\Http\Request的capture方法捕获所有http请求，作为参数传给kernel的handle方法，黑箱处理后$kernel将返回值赋给$response

$response通过send方法将返回传送给浏览器，一个HTTP请求至此结束。

最后kernel调用terminate方法，主要是为了调用中间件的terminate方法做一些扫尾工作。
