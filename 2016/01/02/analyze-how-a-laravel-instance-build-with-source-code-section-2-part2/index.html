结合源码分析Laravel实例$App是怎么一步步构建的(下)[part 2]

发表于 2016-01-02 | 分类于 laravel ， php

在结合源码分析Laravel实例$App是怎么一步步构建的(上)[Part 2]中我们已经讲到了Container的make函数如何解决嵌套依赖最后实例化$abstract的，今天我们先不急着接着讲，先画张图帮助大家理解一下之前的步骤，就没很详细画了


上一次我们已经进行到$this[‘events’]->fire($class=get_class($provider),[$provider])这一步

通过Container实现ArrayAccess接口的offsetGet方法来调用make方法实例化一个对象

回顾到此结束，我们继续研究build方法

我直接在英文注释下解释

可以说build 和make函数就是依赖注入实现的核心了，通过反射方法我们最终拿到了实例方法

这么久的讲解我们终于算是把$this[‘events’]弄明白了

现在再来看
$this['events']->fire($class = get_class($provider), [$provider]);
接着调用fire函数

fire函数怎么来的呢，我觉得有必要把EventServiceProvider注册时函数再贴出来一下

我们把注意力放到第一个return上，闭包函数作为concrete的实参返回了Dispatcher实例调用setQueueResolver函数后的返回值，稍微有点绕，但是大家肯定能看懂，当然后面的闭包作为setQueueResolver的参数我们不多谈，主要是setQueueResolver的返回值，其实基本也能猜出这是个链式调用，返回的是Dispatcher的$this指针

所以fire函数就是Dispatcher的成员函数，我们看看

可以看出fire函数主要干了一件事那就是调用事件之前注册的事件监听函数，写过Android或者桌面应用的同学应该都很清楚地记得事件、监听、广播是用来干什么的，在laravel里面这些东西干的事情也都是一样的，举个简单的例子，比如网站注册了新的用户，不少网站就会在首页滚动条的形式通知欢迎某某新用户加入什么的，这个例子里面新用户注册成功就是个事件，而在首页显示欢迎通告则是个事件监听实现。

回到我们laravel源码分析中，当前的事件则是EventServiceProvider和RoutingServiceProvider实例,但是这两个event并没有绑定事件监听器，当然我们的分析并没有白费，fire函数的调用实在不能算少

fire完了，整个调用回到markAsRegistered中来，最主要的mark功能还没完成呢，很简单

将$provider放入serviceProviders数组中以及将loadedProviders数组$class索引置为true

好了，至此整个$app的构建已经完成了一大半了

现在走到Application构造函数的第三行
$this->registerCoreContainerAliases();

$aliases数组放了laravel核心类的别名和全称，将映射关系通过迭代调用alias函数放到aliases数组中

现在大家应该明白了为什么通过app(‘requesst’)能很方便地拿到request的实例，就是这样来滴

构造函数还剩最后一行了

我就直接贴出调用的主要代码了，相信通过之前的分析，上面的代码实现过程已经了然于心了
主要就是完成路径相关的配置，我也就不细说了，至此一个laravel实例已经构建完毕，人肉运行确实很痛苦

回顾下核心内容，laravel是怎么解决依赖注入的，Container容器扮演了个怎样的角色，怎么通过反射拿到实例，并解决嵌套依赖

框架本身多说两句自己的看法

不少程序员一直纠结语言以及框架的选择，其实没有最好的只有最适合的，有人说php慢，laravel慢，这个我承认，但我觉得

框架最重要的地方还是在于高效构建程序，并具有安全性，可靠性以及可扩展性，至于速度问题，瓶颈主要还是在于IO上，做好各级缓存，框架运行那点差异对大多数项目而言真心不大，至于语言呢，PHP是世界上最好的语言这个梗也是听到耳朵起茧，作为一家互联网公司，永远牢记，人力成本，机器成本，开发速度你只能取其二，至于怎么选那是你的事了

程序员当然也要提高自身的知识水平，不要吊死在一个语言框架上，懂不懂啊 手动斜眼
