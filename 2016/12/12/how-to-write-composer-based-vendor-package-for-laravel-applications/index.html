为什么要编写基于composer的项目包

对laravel框架而言，所有的依赖项都在vendor下，包括laravel framework本身，我认为基于composer的好处有以下几种

  1. 统一规范自动加载规范PSR4。有看过我laravel源码分析的同学应该还记得，laravel框架启动的时候是会require vendor目录下面的的autoload.php的，稍微对composer有一丢丢了解的同学也知道这玩意就是composer生成用来自动加载的。当然不是所有的composer包都是基于PSR4规范的。也有不少包还在走PSR0规范，但是PSR4作为PSR0的补充版本我们应该遵循最新的加载规范。ref: PSR4 Autoloader
  2. 基于composer我们可以将包用在多个框架里不仅仅是laravel框架，最新的TP也添加了对composer的支持 -_- |
  3. composer的版本控制开箱可用，了解composer的同学应该知道我说的是什么。
{% codeblock lang:php %}
    /*
    |--------------------------------------------------------------------------
    | Register The Composer Auto Loader
    |--------------------------------------------------------------------------
    |
    | Composer provides a convenient, automatically generated class loader
    | for our application. We just need to utilize it! We'll require it
    | into the script here so that we do not have to worry about the
    | loading of any our classes "manually". Feels great to relax.
    |
    */
    
    require __DIR__.'/../vendor/autoload.php';
    {% endcodeblock %}

为什么要写一个包呢

  1. 一处编写多处使用，降低开发成本
  2. 需求有变动的时候也只用修改一处，降低维护成本
